# Coding Conventions & Rules

## ✅ DO

### Architecture
- Follow Clean Architecture layer boundaries strictly
- Keep domain entities rich with behavior (Rich DDD)
- **Prioritize purity AND performance** — entities should have zero external dependencies
- Define repository interfaces in the `Domain` layer
- Implement repositories in the `Infrastructure` layer
- Use CQRS pattern for all operations (Command/Query separation)
- Inherit all CQRS handlers from `BaseHandler`
- Use Minimal API endpoint modules for new endpoints
- It is acceptable to expose some business logic into the handler **only when performance requires it** — but default to keeping logic in domain entities

### ID Generation
- **Always use `IdGenerator.New()` to generate new GUIDs** (uses UUID v7 for time-ordered IDs)
- IDs are assigned in code, NOT auto-generated by the database
- Every entity gets its ID via `IdGenerator.New()` at creation time

### Code Style
- Follow .editorconfig rules (enforced at build time via `EnforceCodeStyleInBuild`)
- Use `dotnet format` before committing (automated via Husky pre-commit hook)
- Build must pass with `TreatWarningsAsErrors=true` (enforced on pre-push)
- Use meaningful names for classes, methods, and variables
- Use PascalCase for public members, camelCase for private fields
- Prefix private fields with underscore (`_fieldName`)

### Entity Design
- Use factory methods or private constructors for entity creation
- Encapsulate collections (expose as `IReadOnlyCollection`, mutate via methods)
- Keep validation logic inside entities
- Use Value Objects for composite values
- **Every new entity MUST have a corresponding EF Core configuration file** in `Infrastructure/Configurations/` to define relationships and database constraints

### Database
- Always use Fluent API configuration (in `Infrastructure/Configurations/`)
- **One configuration class per entity** — no exceptions
- Use EF Core Code-First migrations
- Migration commands via Makefile: `make migrate name=MigrationName`
- Define all relationships, indexes, and constraints in the configuration file
- **After any entity/relationship change, UPDATE `conductor/db-diagram.md`** with the new entity, properties, and relationships

### API Design
- Return standardized `ApiResponse` from all endpoints
- Use middleware for cross-cutting concerns (exception handling, auth)
- Use FluentValidation for request validation in the WebAPI layer

## ❌ DON'T

### Architecture
- **DO NOT** reference Infrastructure or WebAPI from Domain or Application layers
- **DO NOT** put business logic in handlers/endpoints — it belongs in Domain entities (unless performance demands it)
- **DO NOT** use anemic entity models (entities with only properties and no behavior)
- **DO NOT** bypass the domain model for database operations
- **DO NOT** use `DateTime.Now` or `DateTime.UtcNow` directly (use banned API analyzer)
- **DO NOT** add external dependencies to domain entities — they must remain pure

### ID Generation
- **DO NOT** use `Guid.NewGuid()` — always use `IdGenerator.New()` (UUID v7)
- **DO NOT** configure database auto-generation of IDs
- **DO NOT** add `ValueGeneratedNever()` in entity configurations — it's handled in ApplicationDbContext

### Code Quality
- **DO NOT** commit code that has warnings (pre-push hook will fail)
- **DO NOT** skip `dotnet format` (pre-commit hook runs it automatically)
- **DO NOT** use APIs listed in `BannedSymbols.txt`

### EF Core
- **DO NOT** add new entities to loaded collections without being aware of change tracking implications
- **DO NOT** use Data Annotations for EF Core configuration — use Fluent API only
- **DO NOT** put migration files in any project other than Infrastructure
- **DO NOT** create an entity without its corresponding configuration file
